Two Approaches

Both solve all 8 issues. The difference is architecture scope.

A

Incremental Enhancement (Recommended)

Fix each issue within existing component structure. Add new pieces surgically.

Changes:
types.ts — extract SyntaxColorKey, ColorEditTarget types
ThemeCustomizer/index.tsx — golden ratio positioning, 2-col grid, toggle language, checkbox green
PanelBody.tsx — add tap + long-press gesture on color dots
ActiveThemeHeader — enlarge dots, add labels on mobile
New: QuickColorPicker.tsx — inline popover for long-press editing
New: hooks/useColorEditGesture.ts — tap vs long-press handler

Pros

  • Minimal blast radius (~6 files)
  • Reuses CompactColorRow, WORD_TYPE_LABELS
  • Each fix independently verifiable
  • Preserves playful spring animations

Cons

  • Color edit logic slightly spread across files
  • ThemeCustomizer file grows (already large)
B

ColorEditSystem Module

Extract a centralized color editing system. Single source of truth for all color interactions.

Changes:
New: types/colorTypes.ts — all color-related types
New: components/ColorEdit/ColorDotButton.tsx — unified dot component
New: components/ColorEdit/QuickColorPicker.tsx — inline popover
New: components/ColorEdit/ResponsiveColorGrid.tsx — 1/2-col layout
New: hooks/useColorEditSystem.ts — centralized state + gestures
ThemeCustomizer — decompose, use new system
PanelBody — use ColorDotButton
ActiveThemeHeader — use ColorDotButton

Pros

  • Clean architecture, DRY
  • Color dots behave identically everywhere
  • Easier to test each piece
  • ThemeCustomizer gets smaller

Cons

  • Larger diff (~10 new/modified files)
  • Higher risk of regressions
  • More complex PR review
